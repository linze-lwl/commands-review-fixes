# 预置环境

Agent 任务对运行环境很敏感。每次任务启动后再临时安装系统工具、语言依赖和业务 SDK，会带来三个问题：启动慢、失败点多、版本不可控。对于 Coding Agent、数据分析、文档处理和测试验证类场景，自定义模板是稳定性和成本控制的基础设施。

最佳实践不是“把依赖都塞进镜像”，而是把稳定、复用、低变化的运行环境前置到模板，把用户数据、任务输入和短期凭证保留在沙箱创建之后注入。模板发布后，应立即创建沙箱做冒烟验证，确认工具链可用，再切换业务流量。

## 业务场景

- Coding Agent 需要固定 Python、Node.js、git、编译工具和测试工具版本。
- 数据分析 Agent 需要预装常用科学计算库、表格处理库和绘图库。
- 企业内部 Agent 需要预装业务 SDK、根证书、命令行工具和基础目录结构。
- 多租户任务需要快速启动，不能把时间浪费在重复下载依赖上。

## 方案设计

模板治理建议分成三步：

1. 构建模板：只放稳定依赖、系统工具和通用启动脚本。
2. 冒烟验证：用该模板创建沙箱，检查关键命令、目录、环境变量和业务入口。
3. 版本切换：验证通过后再让业务任务使用新模板；失败时继续使用旧模板。

这套流程让模板变成可发布的工程资产，而不是一次性的 Dockerfile。

## 示例代码

以下示例展示如何用指定模板创建沙箱，并检查 Agent 任务所需的基础工具。生产环境可以把 `PRACTICE_TEMPLATE` 换成自己的模板名称。

```javascript
import { Sandbox } from "e2b";

const template = process.env.PRACTICE_TEMPLATE || "code-interpreter-v1";
const sandbox = await Sandbox.create(template, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "template-smoke" },
});

try {
  const checks = [
    ["python", "python3 --version"],
    ["git", "git --version"],
    ["workdir", "test -d /tmp && echo tmp-ok"],
  ];

  const results = [];
  for (const [name, command] of checks) {
    const result = await sandbox.commands.run(command, { timeoutMs: 30_000 });
    if (result.exitCode !== 0) {
      throw new Error(`${name} check failed: ${result.stderr || result.stdout}`);
    }
    results.push({ name, output: result.stdout.trim() || result.stderr.trim() });
  }

  console.log({ template, checks: results });
} finally {
  await sandbox.kill();
}
```

## 构建建议

构建自定义模板时，建议遵守以下边界：

- 适合放入模板：语言运行时、包管理器、稳定依赖、基础 CLI、业务 SDK、非敏感配置。
- 不适合放入模板：用户数据、任务输入、生产密钥、短期 Token、高频变化的业务状态。
- 镜像版本应尽量固定。生产环境不建议依赖会漂移的 `latest`。
- 每个模板应有最小冒烟测试，验证后再发布给业务任务使用。

## 可运行示例

本实践对应示例项目中的 `npm run template-smoke`。默认会验证 `code-interpreter-v1`，也可以通过 `PRACTICE_TEMPLATE` 指定自定义模板。
