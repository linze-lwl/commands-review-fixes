# 使用 Computer Use Sandbox (邀测)

> 该部分功能正在内部邀请测试中，内容可能会根据测试情况进行修改，请注意。

Computer Use Sandbox 面向“让 Agent 像人一样操作计算机”的场景，适合需要桌面环境、浏览器或应用窗口、屏幕观察、鼠标键盘输入和结果判定的任务。和 Browser Use Sandbox 不同，Computer Use 不是只控制网页，而是控制一个隔离的 Linux 桌面会话，可以同时操作浏览器、终端、文件管理器、编辑器和业务应用。

根据当前参考实现，正确的使用姿势不是直接把一个无限制桌面交给模型，而是先准备包含 XFCE 桌面、浏览器、截图、鼠标键盘控制、文件和命令能力的 `desktop` 模板，再由业务侧的 Agent Harness 建立“观察、决策、受限执行、审计”的闭环。

如果任务只涉及网页访问、点击、截图和下载，优先使用 [Browser Use Sandbox](02.使用%20Browser%20Use%20Sandbox.md)。如果浏览器操作后还要在同一个会话里运行 Python/Node 处理数据，优先使用 [AIO Sandbox](03.使用%20AIO%20Sandbox.md)。只有任务确实需要跨桌面应用、终端、文件系统或人工远程接管时，才使用 Computer Use Sandbox。

## 业务场景

- 自动操作企业后台，完成跨页面录入、截图、导出文件。
- 复现用户提交的 UI 问题，并生成操作轨迹。
- 让 Agent 在隔离环境中完成浏览器、终端和文件处理的组合任务。
- 对桌面操作 Agent 做评测，记录每一步观察、动作和结果。
- 在远程桌面中让人工接管登录、验证码或高风险确认，再交还给 Agent 继续执行。

## 推荐架构

Computer Use 应按以下方式接入：

1. 业务系统定义任务、允许访问的应用、允许访问的域名和操作边界。
2. 使用已准备好的 `desktop` 模板创建沙箱。该模板通常基于 AIO 自定义运行时，内置 XFCE、浏览器、桌面控制服务和必要工具。
3. Agent Harness 等待桌面就绪，获取屏幕尺寸和初始截图。
4. 将截图、任务目标和历史动作传给视觉模型，由模型返回受限动作。
5. Harness 校验动作，只允许执行明确的 `click`、`double_click`、`right_click`、`type/write`、`press`、`scroll`、`drag`、`wait`、`screenshot` 等动作。
6. 登录、提交表单、付款、删除文件、发送消息、修改配置等高风险动作进入人工确认。
7. 每一步执行后截图，记录观察、动作、窗口状态、模型输出和结果文件。
8. 任务结束后导出轨迹、截图和产物，并销毁沙箱。

## 准备桌面模板

当前 Computer Use 参考实现使用桌面模板，而不是普通 browser 模板。模板应至少具备：

- Ubuntu 22.04 或等价 Linux 基础环境。
- XFCE 等桌面环境。
- 浏览器、终端、文件管理器、编辑器等需要被操作的应用。
- 桌面控制服务，支持截图、鼠标、键盘、窗口、文件和命令能力。
- 可选的远程桌面或 livestream 能力，用于调试和人工接管。

开发和评测阶段可以使用预置或内部预览镜像创建 AIO 自定义运行时模板。生产环境应固化为受控模板，不把账号凭证、Cookie、长期 Token 写入镜像。

## 示例代码

以下示例展示 Desktop SDK 的基本控制闭环：创建桌面沙箱、等待就绪、截图、执行受限动作、再次截图并销毁沙箱。实际接入视觉模型时，把 `screenshot`、屏幕尺寸和历史动作传给模型，由模型返回下一步动作。示例中的 SDK 包名以实际发布版本为准。

```python
import logging
import os
from pathlib import Path

from your_sandbox_sdk import DesktopSandbox

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

TEMPLATE_NAME = os.getenv("SANDBOX_TEMPLATE_NAME") or os.getenv("TEMPLATE_NAME")


def save_screenshot(desktop: DesktopSandbox, path: str) -> None:
    image = desktop.screenshot()
    Path(path).write_bytes(image)
    logger.info("saved screenshot: %s (%s bytes)", path, len(image))


desktop = DesktopSandbox.create(
    template_name=TEMPLATE_NAME,
    wait_timeout=120,
    idle_timeout_seconds=600,
)

try:
    logger.info("sandbox created: %s", desktop.sandbox_id)

    health = desktop.check_health()
    logger.info("health: %s", health)

    width, height = desktop.get_screen_size()
    logger.info("screen size: %sx%s", width, height)

    save_screenshot(desktop, "01_desktop_ready.png")

    # 实际生产中应由视觉模型基于截图返回动作。
    # 这里用固定动作演示 Computer Use 的观察-动作-观察闭环。
    desktop.launch("xfce4-terminal")
    desktop.wait(2000)
    save_screenshot(desktop, "02_terminal_opened.png")

    desktop.write("echo hello from computer use sandbox")
    desktop.press("enter")
    desktop.wait(1000)
    save_screenshot(desktop, "03_command_result.png")

finally:
    desktop.kill()
```

安装和运行时，业务侧需要配置访问凭证和模板名称：

```bash
pip install <desktop-sdk-package> python-dotenv

export FC_ACCESS_KEY_ID=your-access-key-id
export FC_ACCESS_KEY_SECRET=your-access-key-secret
export FC_ACCOUNT_ID=your-main-account-id
export TEMPLATE_NAME=your-desktop-template

python example_desktop.py
```

## 远程观察和人工接管

如果需要让开发者或用户实时观察桌面，可以启动桌面流。生产环境必须开启认证，不要暴露未授权桌面。

```python
from your_sandbox_sdk import DesktopSandbox

desktop = DesktopSandbox.create(
    template_name="your-desktop-template",
    wait_timeout=120,
    idle_timeout_seconds=600,
)

try:
    desktop.stream.start(require_auth=True)
    auth_key = desktop.stream.get_auth_key()
    url = desktop.stream.get_url(auth_key=auth_key)
    print(f"Desktop stream: {url}")

    # 在这里执行 Agent 任务，或等待人工完成登录、验证码、高风险确认。
    desktop.wait(60_000)

finally:
    try:
        desktop.stream.stop()
    finally:
        desktop.kill()
```

如果模板没有提供 `stream` 能力，也可以在模板内启动 VNC 或 noVNC 服务，并通过沙箱端口暴露访问地址。无论使用哪种方式，都应加认证、限时有效并记录访问日志。

## 动作空间设计

建议把模型输出约束为结构化动作，而不是让模型直接生成系统命令：

```json
{
  "action": "click",
  "x": 320,
  "y": 240,
  "reason": "focus the terminal input area"
}
```

Harness 在执行前应做校验：

- 坐标必须在当前屏幕尺寸内。
- 文本输入要限制长度，并过滤明显危险内容。
- `press` 只允许白名单快捷键，例如 `enter`、`escape`、`tab`、`ctrl+a`、`ctrl+c`。
- `commands.run` 只能用于业务侧明确授权的诊断或准备步骤，不作为模型自由动作。
- 每轮动作后必须截图并记录执行结果。

## 上线建议

- 不接受模型生成的任意 Shell 命令，桌面动作和命令执行要分权管理。
- 截图可能包含敏感信息，传给模型前应做脱敏或访问控制。
- 登录态、Cookie、账号凭证应按任务临时注入，不写入模板。
- 登录、支付、删除、发送消息、修改配置等动作必须进入人工确认。
- 记录每一步观察、动作、截图、窗口状态、模型输出和产物，便于复盘和审计。
- 设置任务超时、最大步数、空闲超时和自动清理策略。
- 桌面任务资源成本更高，应只在 Browser/AIO 无法满足时使用 Computer Use。
