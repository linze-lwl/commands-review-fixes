# 休眠与恢复

## 活跃、浅休眠、深休眠分别是什么？

活跃状态表示 Sandbox 正在运行，适合持续执行命令、保持服务端口、处理交互式任务或承载 longrun 任务。

浅休眠用于在保留 Sandbox 现场的同时降低资源占用，并支持快速唤醒。它适合 Agent 对话间隔、代码解释器等待下一次输入、自动化任务阶段性暂停等场景。

深休眠用于在更低资源成本下保留 Sandbox 状态，适合更长间隔的任务恢复、用户稍后继续、长链路任务阶段性挂起等场景。

## 浅休眠和深休眠解决什么问题？

它们同时解决启动加速和资源降本两个问题。

如果每次任务都重新创建 Sandbox，开发者需要反复等待环境准备、依赖加载、服务启动和上下文恢复。休眠/恢复让 Sandbox 可以在保留现场的基础上重新进入可用状态，减少重复初始化。

同时，休眠状态降低了空闲阶段的资源占用，适合交互式 Agent、代码解释器、浏览器/工具执行链路等“有状态但不是一直计算”的场景。

## 浅休眠唤醒有多快？

云沙箱浅休眠可以做到毫秒级休眠唤醒，适合对交互延迟敏感的任务，例如 Agent 连续工具调用、代码解释器多轮执行、Web 服务预览中的短间隔访问。

开发者可以把浅休眠理解为面向高频恢复的状态保持能力：既保留执行现场，又尽量降低下一次调用的等待时间。

实际恢复耗时会受任务状态、资源规格、网络访问和平台调度等因素影响。建议在业务链路中结合健康检查和超时重试机制评估端到端恢复时延。

## 深休眠唤醒适合什么场景？

云沙箱深休眠具备约 1s 级别的恢复时效，适合对恢复速度有要求、但不需要毫秒级唤醒的场景。

例如用户稍后继续任务、Coding Agent 等待人工确认、数据处理任务分阶段推进、长链路自动化任务中间暂停。相比重新创建 Sandbox，深休眠可以保留现场并降低重复初始化成本。

## 云沙箱休眠/恢复和 E2B pause/resume 有什么区别？

E2B 的 pause/resume 主要面向 Sandbox 暂停与恢复的开发者体验：暂停一个 Sandbox，后续再连接同一个 Sandbox 继续使用。

云沙箱在兼容类似开发体验的基础上，把休眠能力进一步产品化为活跃、浅休眠、深休眠等生命周期状态。开发者不只是在“暂停/恢复”之间切换，还可以根据恢复速度、资源成本和任务间隔选择更合适的状态。

浅休眠强调毫秒级恢复和交互体验；深休眠强调更低资源占用和秒级恢复。二者共同服务 longrun、有状态 Agent、代码解释器和沙箱内服务等场景。

## 开发者应该如何选择浅休眠和深休眠？

如果任务很快会继续，或者下一次调用对延迟敏感，优先使用浅休眠。

如果任务间隔更长，希望进一步降低空闲资源成本，同时可以接受约 1s 级别恢复，优先使用深休眠。

如果任务已经完成，不再需要恢复现场，应终止 Sandbox，释放资源。

## 使用休眠/恢复时需要记录什么？

至少记录 `sandboxId`、任务 ID、模板版本、当前阶段、关键文件路径、关键进程或端口信息。

恢复后建议做一次轻量健康检查，例如确认命令通道可用、关键文件存在、服务端口可访问。这样可以把休眠/恢复纳入稳定的业务状态机，而不是只依赖一次 SDK 调用结果。
