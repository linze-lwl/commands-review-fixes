# 使用场景

## 什么时候应该使用云沙箱？

当任务需要运行代码、执行命令、处理文件、启动沙箱内服务，且需要隔离执行环境时，使用云沙箱。

典型场景包括 Coding Agent 跑测试、代码解释器、数据文件处理、自动化脚本、Web 服务预览、执行用户或模型生成的代码。核心判断标准是：任务需要隔离、可编排生命周期、文件/进程状态，或需要在 longrun、休眠、恢复之间保持任务现场。

## 云沙箱是否支持长时间运行和状态保持？

支持。云沙箱不是只能跑短任务的一次性执行容器。对于 Agent、代码解释器、自动化测试、Web 服务预览等场景，开发者可以让 Sandbox 保持运行，也可以通过休眠/恢复保留任务现场。

业务侧仍应记录 `sandboxId`、任务 ID、模板版本和关键状态，便于恢复、排查和清理。

## 已有 E2B 应用应该怎么迁移？

先保持 SDK 调用方式不变，只替换 API Key、API URL 和域名。再逐项核对能力边界，详情参见[E2B 兼容与迁移](02.E2B%20兼容与迁移.md)。

优先验证四件事：创建 Sandbox、执行命令、读写文件、释放资源。依赖 Snapshot、Volume、Team、Access Token 管理、Logs、高频 Metrics 或 Network Config Update 的应用，需要先调整设计。

## 云沙箱适合生产环境吗？

适合，但要按远端执行资源设计：可失败、可重试、可释放、可观测。

生产代码至少要记录 `sandboxId`、模板版本、任务 ID、创建时间和关键错误；明确 longrun、休眠、恢复、终止策略；任务结束后主动释放资源；需要跨 Sandbox 或跨业务流程使用的数据应写入外部存储。
