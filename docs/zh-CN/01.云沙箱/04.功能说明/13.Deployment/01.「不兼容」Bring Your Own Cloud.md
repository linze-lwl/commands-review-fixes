# 「不兼容」自带云部署（Bring Your Own Cloud）

Bring Your Own Cloud（BYOC，自带云部署）是 E2B 平台的一种部署形态，指把 E2B 的控制面或运行面部署到用户自己的云账号、私有云或 Kubernetes 集群中。

云沙箱不支持 E2B BYOC。云沙箱是阿里云函数计算提供的云产品能力，开发者通过云沙箱的 Endpoint、API Key、模板和 Sandbox 管理能力接入，不需要也不能自行部署云沙箱控制面。

## 什么时候会遇到这个差异

从 E2B 迁移时，如果应用或运维文档提到以下内容，通常是在使用 E2B BYOC：

- 在用户自己的云账号中部署 E2B 控制面。
- 在自建 Kubernetes 集群中运行 E2B 基础设施。
- 自行管理 E2B 的运行面、调度、镜像仓库、网络入口或集群升级。
- 通过 E2B Deployment 配置把平台部署到指定云环境。

这些能力不属于云沙箱兼容范围。迁移时不要把 E2B BYOC 部署步骤直接套到云沙箱上。

## 常见替代路径

很多“看起来像 BYOC”的诉求，其实只是希望 Sandbox 访问企业内网、使用固定域名、挂载数据或接入可观测能力。云沙箱用 FC Extensions 处理这些场景：

- 让 Sandbox 访问 VPC 内的数据库、内网 API 或镜像仓库：使用 [VPC 网络配置](../07.FC%20Extensions/02.VPC%20网络配置.md)。
- 使用固定域名和证书访问云沙箱 API 或 Sandbox 内服务：使用 [自定义域名](../07.FC%20Extensions/03.自定义域名.md)。
- 将 OSS 数据挂载到 Sandbox 内：使用 [动态挂载 OSS](../07.FC%20Extensions/04.动态挂载%20OSS.md)。
- 查看运行日志、资源指标或接入日志服务：使用 [监控与日志](../07.FC%20Extensions/05.监控与日志.md)。

这些能力属于云沙箱的云上扩展，不属于 E2B Deployment / BYOC。

## 迁移建议

迁移依赖 BYOC 的 E2B 应用时，先拆分真实诉求，再选择路径：

- 需要私网访问、域名、对象存储或日志监控：按 FC Extensions 逐项迁移。
- 需要数据驻留或合规隔离：选择合适的阿里云地域，配置 VPC、权限、日志和存储策略。
- 需要把沙箱控制面完整部署到自己的云账号或私有集群：当前不属于云沙箱兼容范围，请联系产品支持确认是否有企业级交付方案。

应用代码侧通常不需要感知 BYOC。保留 E2B SDK 的调用方式，替换云沙箱接入参数，再按需配置云上扩展即可。
