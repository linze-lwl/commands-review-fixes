# Resources and Costs

## What if concurrency is insufficient?

First determine whether the limit comes from sandbox concurrency, template build concurrency, account quota, regional capacity, or rate limiting in your own business system.

Do not build templates in real time on the online request path. For batch workloads, define queues, concurrency caps, timeouts, and backoff retries. If you need higher quota, apply through the Function Compute console or Alibaba Cloud support channels.

## Do paused sandboxes consume concurrency or incur charges?

Follow the current FC Agent Sandbox billing and quota rules. Developers need to distinguish running, hibernating, resuming, and terminated states instead of mixing lifecycle states together.

Use hibernation and resume when the task will continue and the runtime state should be preserved. Call `kill()` to release resources when the task is already complete.

## Why should I release sandboxes explicitly?

Sandboxes consume compute, memory, disk, ports, and concurrency resources. Explicit release reduces cost and also avoids security risks caused by background processes continuing to run.

Every success, failure, timeout, and cancellation path should enter cleanup logic.

## How can I reduce repeated initialization costs?

Put stable dependencies into templates and keep task inputs at runtime. Do not reinstall large dependencies after every sandbox creation.

Common optimizations include pinning image versions, pinning dependency versions, preinstalling common CLI tools, reusing templates, and reducing runtime downloads.

## How should longrun tasks be designed?

Longrun tasks should manage the lifecycle around `sandboxId`, not only around the return value of a single command.

It is recommended to record the template version, startup command, key ports, health check method, and business task ID. After reconnecting, confirm the command channel, critical processes, and file state before continuing execution.

## Can files be stored in a sandbox?

Yes. The sandbox filesystem can be used to store task input, intermediate files, and runtime state, and it can continue to be used after hibernation and resume.

If data must be shared across sandboxes or across business workflows, or retained for auditing, write it to OSS, NAS, a database, or business storage.

## How can I detect resource leaks?

List running sandboxes regularly and inspect unusually long-lived instances by task ID, user ID, template, and creation time.

Production wrappers should record create and release events, and cleanup paths for timeouts, cancellations, and exceptions should be covered by tests.
