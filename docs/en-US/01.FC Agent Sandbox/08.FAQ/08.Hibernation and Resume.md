# Hibernation and Resume

## What are active, light hibernation, and deep hibernation?

The active state means the sandbox is running. It is suitable for continuous command execution, keeping service ports open, handling interactive tasks, or carrying longrun workloads.

Light hibernation preserves sandbox state while reducing resource usage and supports fast wake-up. It is suitable for agent conversation gaps, waiting for the next code interpreter input, and phase-based pauses in automated tasks.

Deep hibernation preserves sandbox state at a lower resource cost. It is suitable for longer task gaps, users resuming later, and staged suspension in long-running workflows.

## What problems do light and deep hibernation solve?

They solve both startup acceleration and resource cost reduction.

If every task recreates a sandbox from scratch, developers repeatedly wait for environment preparation, dependency loading, service startup, and context restoration. Hibernation and resume allow a sandbox to return to an available state while preserving its runtime context, which reduces repeated initialization.

At the same time, hibernation reduces resource consumption during idle periods. It is suitable for interactive agents, code interpreters, browser or tool execution chains, and other scenarios that are stateful but not continuously computing.

## How fast is light hibernation resume?

FC Agent Sandbox light hibernation can resume at millisecond-level latency. It is suitable for latency-sensitive tasks such as consecutive agent tool calls, multiple rounds in a code interpreter, and short-interval access in web service preview.

Developers can think of light hibernation as a state-preservation capability optimized for high-frequency resume: it preserves runtime context while keeping the next call as fast as possible.

Actual resume latency depends on task state, resource specifications, network access, and platform scheduling. Evaluate end-to-end resume latency in your business flow with health checks, timeouts, and retries.

## What scenarios are suitable for deep hibernation resume?

FC Agent Sandbox deep hibernation provides resume latency at approximately the 1 s level. It is suitable for scenarios where recovery speed matters, but millisecond wake-up is not required.

Examples include users continuing a task later, a coding agent waiting for human confirmation, staged data-processing workflows, and paused intermediate steps in long-running automation. Compared with recreating a sandbox, deep hibernation preserves runtime state and reduces repeated initialization cost.

## How is FC Agent Sandbox hibernation and resume different from E2B pause and resume?

E2B pause and resume primarily focus on the developer experience of suspending and restoring a sandbox: you pause a sandbox and later reconnect to the same sandbox to continue.

FC Agent Sandbox keeps a similar developer experience while turning hibernation into a more fully productized lifecycle model with states such as active, light hibernation, and deep hibernation. Developers are not limited to switching between pause and resume. They can choose a more suitable state based on resume speed, resource cost, and task interval.

Light hibernation emphasizes millisecond-level resume and interactive experience. Deep hibernation emphasizes lower resource usage and second-level resume. Together they support longrun, stateful agents, code interpreters, and services running inside sandboxes.

## How should developers choose between light and deep hibernation?

If the task will continue soon or the next invocation is latency-sensitive, prefer light hibernation.

If the interval is longer and you want to reduce idle resource cost further while still accepting resume latency at roughly the 1 s level, prefer deep hibernation.

If the task is complete and the runtime state no longer needs to be preserved, terminate the sandbox and release resources.

## What should be recorded when using hibernation and resume?

At a minimum, record the `sandboxId`, task ID, template version, current phase, key file paths, and key process or port information.

After resume, it is recommended to run a lightweight health check, such as confirming that the command channel is available, key files exist, and service ports are reachable. This makes hibernation and resume part of a stable business state machine instead of relying only on the result of a single SDK call.
