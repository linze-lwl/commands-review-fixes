# Lifecycle and Connection

## What should I check first when sandbox creation fails?

Check four items first: whether the API key is correct, whether `E2B_API_URL` and `E2B_DOMAIN` are in the same region, whether the template name exists, and whether the account has the required permissions and quota.

It is recommended to run the minimum example first. If the minimum example fails, the issue is usually with authentication, endpoints, region, or quota. If the minimum example succeeds, then investigate the custom template and business command.

## What if creation succeeds but the sandbox becomes unavailable immediately?

Run a minimum command first to confirm the channel is available:

```bash
echo ok
```

If the minimum command works, the issue is most likely in the business command, dependency set, working directory, or permissions. If the minimum command does not work, check the sandbox status, template startup logic, and service-side errors.

## What are common causes of `Sandbox.connect(sandboxId)` failures?

Common causes include a `sandboxId` that does not belong to the current account or region, a sandbox that has already been `kill()`ed, a sandbox reclaimed due to timeout, or an SDK version mismatch.

The business side should persist the `sandboxId`, template, task ID, and creation time. After reconnection, run a lightweight health check instead of assuming that the previous process, port service, and external connections are still usable.

## How should I choose between `pause()` and `kill()`?

Use `pause()` when the task will continue later and the runtime state must be preserved. Use `kill()` when the task is complete, cannot be recovered after failure, or has exceeded the business wait time.

Pause is used to preserve sandbox state so that you can later reconnect to the same `sandboxId`. Do not use `kill()` as if it were pause. After `kill()`, you cannot return to the same runtime state.

## Is state always retained after pause and resume?

Pause and resume are intended to preserve sandbox state. After resume, the business side should still run a lightweight health check to confirm that the command channel, critical files, and service ports are in the expected state.

If state must be retained across sandboxes, across regions, or for long-term auditing, synchronize it to external storage or a business database.

## What happens if I forget to call `kill()`?

The sandbox continues to consume resources until it is terminated manually, times out, or is reclaimed by the platform. Production code should use `try/finally` or an equivalent mechanism to release resources.

```typescript
try {
  await sandbox.commands.run("python3 main.py", { timeoutMs: 120_000 });
} finally {
  await sandbox.kill();
}
```

## How does Sandbox support longrun workloads?

For longrun workloads, define three things clearly: the sandbox lifecycle, how commands or services run, and the business-side recovery strategy.

For agents, code interpreters, automated tasks, or web services inside sandboxes that need to keep running, persist the `sandboxId`, record key processes and ports, and run a health check after reconnecting. Call `kill()` to release resources only after the task is complete.
