# Use Cases

## When should I use FC Agent Sandbox?

Use FC Agent Sandbox when a task needs to run code, execute commands, process files, or start services inside a sandbox, and an isolated execution environment is required.

Typical scenarios include coding agents running tests, code interpreters, data file processing, automation scripts, web service preview, and execution of user-generated or model-generated code. The core decision criteria are whether the task needs isolation, an orchestrated lifecycle, file or process state, or the ability to preserve task state across longrun, hibernation, and resume.

## Does FC Agent Sandbox support long-running tasks and state retention?

Yes. FC Agent Sandbox is not limited to one-shot execution containers for short tasks. For agents, code interpreters, automated testing, web service preview, and similar scenarios, developers can keep a sandbox running or preserve task state through hibernation and resume.

The business side should still record the `sandboxId`, task ID, template version, and key state so that recovery, troubleshooting, and cleanup remain manageable.

## How should existing E2B applications migrate?

Keep the SDK invocation pattern unchanged first, and replace only the API key, API URL, and domain. Then verify capability boundaries item by item. For details, see [E2B Compatibility and Migration](02.E2B%20Compatibility%20and%20Migration.md).

Validate four things first: sandbox creation, command execution, file reads and writes, and resource release. Applications that depend on Snapshot, Volume, Team, access token management, logs, high-frequency metrics, or Network Config Update need a design adjustment first.

## Is FC Agent Sandbox suitable for production?

Yes, but it should be designed as a remote execution resource: it can fail, be retried, be released, and be observed.

At a minimum, production code should record the `sandboxId`, template version, task ID, creation time, and key errors; define its longrun, hibernation, resume, and termination strategy; proactively release resources after the task finishes; and write data that must outlive a sandbox or a business step into external storage.
