# Network and Service Access

## What if I cannot access a web service started in a sandbox?

First confirm that the service is listening on the target port inside the sandbox and that the process has not exited. Then use `sandbox.getHost(port)` to retrieve the access address.

`getHost(port)` returns the host. In most cases, you need to prepend `https://` when accessing it. Do not construct internal addresses yourself from debugging fields.

## Why does the public URL still return 502 or timeout after the background service starts?

Common causes include the service not being ready yet, listening on the wrong port, listening on an address other than `0.0.0.0`, exiting after startup, or failing during dependency initialization.

Check service logs and port listening first, and then access the result of `getHost(port)`. If the service needs startup time, the business side should perform bounded retries instead of failing immediately.

## How do I troubleshoot public internet access failures inside a sandbox?

Run a minimum network check inside the sandbox first so you can distinguish DNS failures, connection timeouts, TLS failures, HTTP status code issues, and restrictions imposed by the destination site.

External networks should not be treated as stable resources. Dependency installation, web crawling, and third-party API access should all include timeouts, retries, and failure handling.

## How do I access cloud resources such as VPC, RDS, OSS, and NAS?

This is not purely an SDK issue. It involves both network paths and permission boundaries.

First confirm the access path: public internet, VPC, PrivateLink, internal endpoint, or forwarding through a business service. Then confirm permissions: RAM role, temporary credentials, least privilege, and secret rotation.

Do not write long-lived AccessKeys into templates, environment variables, command lines, logs, or result files.

## Can OSS, S3, or R2 be mounted directly to a sandbox?

FC Agent Sandbox can mount OSS, NAS, storage from other cloud vendors, or self-managed storage into a sandbox so that the sandbox can directly access external datasets, model files, task output directories, or existing enterprise storage systems.

This capability is currently in Beta. Support scope, network environments, permission models, performance characteristics, and enablement methods may differ by storage type. It is recommended to contact the product or sales team to confirm the supported scope and integration plan.

If you only need small-file input and output for a one-time task, you can continue to use the Filesystem API to write input files and retrieve results after processing completes.

## What are the risks of exposing sandbox ports publicly?

Do not expose sensitive data, internal debugging interfaces, management entry points, or long-lived credentials through unauthenticated ports.

Port services should serve only the current task. After the task completes, terminate the background process or release the sandbox.
