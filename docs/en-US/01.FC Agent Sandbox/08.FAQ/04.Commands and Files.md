# Commands and Files

## What if a command has no output?

Check stdout, stderr, and the exit code together first. Many tools write errors to stderr, write logs to files, or continue running as background processes.

```typescript
const result = await sandbox.commands.run("python3 script.py");
console.log(result.exitCode);
console.log(result.stdout);
console.error(result.stderr);
```

## Does a non-zero command exit code always mean an FC Agent Sandbox issue?

No. A non-zero exit code usually means the command itself failed. Common causes include missing dependencies, incorrect paths, insufficient permissions, network failures, or invalid arguments.

Run a minimum command with the same template first, then add business dependencies gradually. Do not attribute every failure directly to sandbox creation.

## When do I need PTY?

Use PTY only when a command depends on terminal behavior, such as an interactive CLI, TTY detection, colored output, progress bars, or commands that require ongoing input.

For normal batch processing, testing, builds, and script execution, PTY should not be enabled by default because stdout and stderr parsing becomes less stable.

## Why can I not access a service after starting a background command?

Common causes are that the service is not listening on the target port, exits immediately after startup, listens on the wrong address, or has not finished initializing its dependencies.

Check ports and logs inside the sandbox first, and then call `sandbox.getHost(port)`. Do not assume the service is ready just because the background command returned success.

## How do I troubleshoot a file that a command cannot read after it is written?

Check the path, ordering, working directory, and permissions. For production tasks, use absolute paths and create a dedicated directory for each task, such as `/tmp/tasks/<task-id>`.

If multiple executions involve different sandboxes or different execution contexts, also confirm that the file write and the command execution happened in the same environment.

## Can upload and download URLs be stored long term?

No. Upload and download URLs should be treated as temporary addresses. Do not write them into long-lived logs, tickets, frontend pages, or any location users can access persistently.

For long-term file retention, write the files to OSS, NAS, a database, or business storage.

## What should I check first when `run_code` fails?

First confirm that you are using a Code Interpreter template such as `code-interpreter-v1`. Then check whether Python, the working directory, and the required dependencies are available.

```bash
python3 --version
which python3
pwd
```

If normal commands work but `run_code` fails, focus on whether the template includes the services and startup logic required by Code Interpreter.

## Are variables retained across multiple executions?

Yes, within the same sandbox and the same execution context. Once you switch sandboxes, switch contexts, or terminate and recreate the sandbox, do not assume variables are still present.

Business results should be written to files or external storage. Do not rely only on in-memory variables inside the interpreter.
