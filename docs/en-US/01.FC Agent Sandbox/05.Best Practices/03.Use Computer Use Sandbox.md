# Use Computer Use Sandbox

Computer Use Sandbox is designed for scenarios where an Agent needs to operate a computer the way a person would. That usually means a desktop environment, a browser or application window, screen observation, clicking, typing, and evaluating results. If you already have a `desktop` template that includes a desktop environment, browser, screenshot support, and input control capabilities, you can follow the E2B Computer Use pattern: create a desktop sandbox, capture screenshots, send those screenshots to a vision model for decisions, and then send actions such as click, type, scroll, and keypress back to the sandbox for execution.

This practice fits desktop automation, complex back-office operations, UI issue reproduction, mixed browser and terminal workflows, and Computer Use Agent evaluation. Compared with Browser Use Sandbox, Computer Use Sandbox is not limited to websites. It can also operate windows, terminals, file managers, and multi-application workflows, so permissions, action boundaries, and human approval need to be controlled more strictly.

## Use cases

- Automate enterprise back-office systems across multiple pages to enter data, capture screenshots, and export files.
- Reproduce UI issues reported by users and generate an action trace.
- Let an Agent complete combined browser, terminal, and file-processing workflows inside an isolated environment.
- Evaluate desktop-operation Agents while recording every observation, action, and result.

## Recommended architecture

Computer Use should not start from the idea of giving the model an unrestricted desktop. A safer architecture is:

1. The business system defines the task, the allowed applications, and the operation boundaries.
2. Create FC Agent Sandbox from a prepared `desktop` template.
3. The Agent Harness captures screenshots and sends observations to a vision model.
4. The model outputs restricted actions such as `click`, `type`, `scroll`, and `keypress`.
5. High-risk actions such as form submission, payment, file deletion, and message sending go through human approval.
6. Export the trace, screenshots, and artifacts at the end of the task, then destroy the environment.

## Example code

The following example mirrors the E2B Computer Use developer experience and assumes the desktop template has already been verified as compatible with `@e2b/desktop`. It creates a `desktop` sandbox, opens a page, takes a screenshot, and then performs one click and one input action. In a real integration, pass `screenshot` to the model and let the model return the next action.

```javascript
import { Sandbox } from "@e2b/desktop";

const desktop = await Sandbox.create({
  template: "desktop",
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  resolution: [1024, 768],
  dpi: 96,
  timeoutMs: 300_000,
  metadata: { scenario: "computer-use" },
});

try {
  await desktop.launch("firefox", "https://example.com");

  const screenshot = await desktop.screenshot();
  // Send screenshot to the vision model and let the model return the next action.
  // This example uses fixed actions to demonstrate the Computer Use control loop.

  await desktop.mouse.click(320, 240);
  await desktop.keyboard.type("hello from sandbox");
  await desktop.keyboard.press("ENTER");

  const after = await desktop.screenshot();
  console.log({
    beforeBytes: screenshot.length,
    afterBytes: after.length,
  });
} finally {
  await desktop.kill();
}
```

If users or developers need to observe the desktop in real time, you can start a remote desktop or VNC or streaming service in the template and use `getHost(port)` to obtain the access URL. In production, that URL must be protected with authentication instead of exposing an unauthenticated desktop.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("desktop", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  metadata: { scenario: "desktop-preview" },
});

const vnc = await sandbox.commands.run("start-vnc --host 0.0.0.0 --port 6080", {
  background: true,
  timeoutMs: 30_000,
});

console.log(`Desktop preview: https://${sandbox.getHost(6080)}`);
console.log(`VNC process id: ${vnc.pid}`);
```

Here `start-vnc` is only an example startup script from the `desktop` template. Use the actual script provided by your template. As long as the service listens on a sandbox port, you can use `getHost(port)` to obtain the access URL.

## Production recommendations

- Restrict the action space to explicit operations such as `click`, `type`, `scroll`, and `keypress`. Do not accept arbitrary system commands generated by the model.
- Screenshots may contain sensitive information. Apply masking or access control before sending them to the model.
- Actions such as login, payment, deletion, message sending, and configuration changes must go through human approval.
- Record every observation, action, screenshot, window state, and model output for review and audit.
- Desktop tasks usually cost more resources, so set timeouts, maximum step counts, and automatic cleanup policies.
