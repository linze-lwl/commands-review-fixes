# Host a Hermes Agent

Hermes Agent is a self-evolving open source Agent from Nous Research. It supports CLI or TUI interaction, message gateways, persistent memory, Skills, MCP, browser automation, terminal tools, sub-agents, and scheduled tasks. According to the official Hermes documentation, Linux, macOS, and WSL2 can install the CLI through an install script. For first-time setup, use `hermes setup` or `hermes setup --portal` to configure the model and tools. For daily interaction, use `hermes`, `hermes --tui`, or `hermes chat -q`.

FC Agent Sandbox is well suited to hosting Hermes as a complete Agent application in an isolated cloud environment. Preinstall the Hermes CLI, Python and Node.js dependencies, browser dependencies, and common tools into a template. After creating a sandbox, start the CLI, TUI, message gateway, or custom service through `commands.run()`. If the template also exposes an HTTP console, a message gateway callback service, or a debug service, obtain its access address through `getHost(port)`.

## Use cases

- A team wants to run Hermes Agent in the cloud instead of on a developer laptop.
- An enterprise wants to evaluate Hermes Skills, MCP, browser automation, and terminal tools in an isolated execution environment.
- The business needs a dedicated Agent environment per customer, project, or evaluation task, with release after the task ends.
- The Agent needs to run commands, process files, access browsers, or call external tools, and the business wants to limit the blast radius through sandbox isolation.

## Recommended architecture

Use a "Hermes template, sandbox instance, and external state" architecture:

- **Custom template**: preinstall the Hermes CLI, Python, Node.js, Playwright or browser dependencies, `ripgrep`, `ffmpeg`, and similar runtime components.
- **FC Agent Sandbox**: create the Hermes runtime per task, user, or project, and use it to run the CLI, TUI, gateway, Skills, and tool calls.
- **External systems**: store user identity, long-term memory, audit logs, model credentials, and business data.

Hermes has its own security mechanisms, such as command approval, YOLO mode warnings, and dangerous command blocking. FC Agent Sandbox provides an outer isolation boundary and lifecycle management. The two should be used together instead of as substitutes.

## Build a Hermes template

The official Hermes install script handles Python, Node.js, `ripgrep`, `ffmpeg`, the virtual environment, and the global `hermes` command. For an FC Agent Sandbox template, the better approach is to create a dedicated image and complete the installation during image build.

```dockerfile
FROM debian:bookworm

RUN apt-get update \
  && apt-get install -y --no-install-recommends \
    ca-certificates curl git xz-utils build-essential \
  && rm -rf /var/lib/apt/lists/*

RUN curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash -s -- --skip-browser

ENV PATH="/root/.local/bin:/root/.hermes/hermes-agent/venv/bin:${PATH}"
WORKDIR /workspace
```

If browser automation is required, do not use `--skip-browser`, and add the Playwright or Chromium system dependencies recommended by Hermes. Browser support is more sensitive to image size, memory, and startup time, so validate it in a dedicated template.

After building and pushing the image, create an FC Agent Sandbox template through the E2B SDK:

```python
import os
import time
from e2b import Template, default_build_logger

image = os.environ["HERMES_IMAGE"]

build = Template.build(
    Template().from_image(image),
    name=f"hermes-agent-{int(time.time())}",
    cpu_count=2,
    memory_mb=4096,
    api_key=os.environ["E2B_API_KEY"],
    api_url=os.environ["E2B_API_URL"],
    domain=os.environ["E2B_DOMAIN"],
    on_build_logs=default_build_logger(),
)

print("template_id:", build.template_id)
```

## Start and validate Hermes

After creating the sandbox, first use the CLI to validate that Hermes works correctly. Initial configuration can be completed with `hermes setup`, `hermes model`, `hermes tools`, or `hermes gateway setup`. In production, convert those steps into initialization scripts and inject model and tool credentials through runtime environment variables.

```typescript
import { Sandbox } from "e2b";

const template = process.env.HERMES_TEMPLATE_ID;
if (!template) {
  throw new Error("HERMES_TEMPLATE_ID is required");
}

const sandbox = await Sandbox.create(template, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: {
    HERMES_HOME: "/workspace/.hermes",
    // Inject model credentials or Nous Portal credentials according to Hermes provider requirements.
    // Do not write real credentials into the template or the source repository.
  },
  metadata: {
    app: "hermes",
    purpose: "agent-demo",
  },
});

try {
  await sandbox.commands.run("mkdir -p /workspace/.hermes");

  const doctor = await sandbox.commands.run("hermes doctor", {
    cwd: "/workspace",
    timeoutMs: 60_000,
  });
  console.log(doctor.stdout);

  const reply = await sandbox.commands.run(
    `hermes chat -q "Inspect the current directory and explain whether this runtime is ready to execute Agent tasks"`,
    {
      cwd: "/workspace",
      timeoutMs: 180_000,
    },
  );
  console.log(reply.stdout);
} finally {
  // In demo environments you can keep the sandbox so users can continue interacting.
  // Release it after automated tests finish.
  // await sandbox.kill();
}
```

If you need an interactive terminal, run `hermes` or `hermes --tui` through PTY. For automated tasks, prefer non-interactive commands such as `hermes chat -q`, `hermes model`, `hermes tools`, and `hermes skills`.

## Access a gateway or custom web service

The official Hermes CLI is a TUI, not a Web UI. FC Agent Sandbox documentation should not imply that Hermes always includes a web console. A safer description is that Hermes itself is used through the CLI or TUI and messaging platforms. If your template additionally starts a Hermes gateway, callback service, custom admin console, or debug service, use `getHost(port)` the same way you would for any other HTTP service inside FC Agent Sandbox.

```typescript
const process = await sandbox.commands.run(
  "start-hermes-gateway --host 0.0.0.0 --port 8080",
  {
    cwd: "/workspace",
    background: true,
    timeoutMs: 30_000,
  },
);

const url = `https://${sandbox.getHost(8080)}`;
console.log(url);
console.log(process.pid);
```

Here `start-hermes-gateway` is only an example startup script from the template, not an official Hermes command. The script can wrap the current Hermes gateway, a messaging callback service, or a team-defined console. Before baking it into a template, verify the current command surface inside the sandbox with `hermes gateway setup`, `hermes --help`, and the service-specific `--help`. As long as the service listens on a sandbox port, you can obtain the access URL through `getHost(port)`.

## Connect to an existing Hermes sandbox

If a Hermes instance must be managed across requests, save `sandbox.sandboxId` and reconnect through `Sandbox.connect()` later:

```typescript
const sandboxId = process.env.HERMES_SANDBOX_ID;
if (!sandboxId) {
  throw new Error("HERMES_SANDBOX_ID is required");
}

const sandbox = await Sandbox.connect(sandboxId, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
});

const sessions = await sandbox.commands.run("hermes --continue --help", {
  cwd: "/workspace",
});

console.log(sessions.stdout);
```

If the sandbox is used only for one-off execution, call `kill()` when the task ends. If the session must be preserved, manage the lifecycle through pause and resume capabilities.

## Best practices

- **Validate the CLI baseline first**: get `hermes doctor` and `hermes chat -q` working before enabling the gateway, Skills, MCP, browsers, or scheduled tasks.
- **Pin the runtime through templates**: keep Hermes, Python, Node.js, browser dependencies, and common tools in the template so you do not reinstall them after every sandbox creation.
- **Externalize state**: Hermes long-term memory, configuration, audit logs, and business data should live outside the sandbox. The local filesystem is suitable for task-level temporary state, not as the only persistence layer.
- **Use YOLO mode carefully**: Hermes supports a no-approval YOLO mode. Consider it only for one-off, disposable, low-risk sandbox tasks, and combine it with outer permission controls.
- **Enable tools gradually**: start with file and terminal tools, then add browsers, MCP, messaging platforms, and scheduled tasks only when needed. Each tool category should come with explicit permission boundaries and log collection.
