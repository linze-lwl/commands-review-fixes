# Build an Agent Harness

An Agent Harness is the middleware that turns a model into an executable system. It is responsible for tool routing, context management, permission approval, trace logging, retry on failure, and result aggregation. Without a Harness, Agent tool calls easily turn into scattered scripts. Without an isolated execution layer, the Harness ends up pushing untrusted code, user files, third-party dependencies, and long-running tasks onto the business service.

FC Agent Sandbox is a good execution layer for an Agent Harness. The Harness decides what to do, whether it is allowed, and how it should be recorded. FC Agent Sandbox decides where it runs, how it is isolated, and how it is reclaimed. That split gives the Agent system both business control and engineering reproducibility.

## Use cases

- An internal enterprise Agent needs to call multiple tools, and every execution must be isolated, auditable, and disposable.
- A Coding Agent needs to run tests inside a repository, generate diffs, and apply fixes.
- A data Agent needs to process uploaded user files and return structured results.
- A multi-tenant Agent platform needs task-level isolation for files, processes, logs, and temporary credentials.

## Solution design

A production-oriented Harness usually contains five modules:

1. Task entry, which receives the user request, tenant identity, task context, and policy configuration.
2. Tool registry, which maintains callable tools, parameter schemas, timeouts, and permission requirements.
3. Sandbox executor, which creates FC Agent Sandbox, writes input files and tool scripts, and runs commands.
4. Trace recorder, which records tool calls, sandbox ID, `stdout` and `stderr`, `exitCode`, and artifact paths.
5. Decision layer, which decides whether to continue, request human approval, or end the task based on execution results.

## Example code

The following example simulates a Harness. It first profiles order data, then runs a business rule check, and finally decides whether automatic import is allowed.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("code-interpreter-v1", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  metadata: { taskId: "task-harness-001" },
});

const tools = {
  profile_dataset: {
    command: "python3 tools/profile_dataset.py input/orders.csv",
    cwd: "/tmp/agent-harness",
    timeoutMs: 30_000,
  },
  validate_business_rule: {
    command: "python3 tools/validate_business_rule.py input/orders.csv",
    cwd: "/tmp/agent-harness",
    timeoutMs: 30_000,
  },
};

try {
  await sandbox.files.makeDir("/tmp/agent-harness");
  await sandbox.files.makeDir("/tmp/agent-harness/input");
  await sandbox.files.makeDir("/tmp/agent-harness/tools");
  await sandbox.files.write(
    "/tmp/agent-harness/input/orders.csv",
    "order_id,customer,amount,status\n1001,Acme,1200,paid\n1002,Acme,900,paid\n1003,Globex,300,refunded\n1004,Initech,0,paid\n",
  );
  await sandbox.files.write(
    "/tmp/agent-harness/tools/profile_dataset.py",
    `import csv, json, sys
rows = list(csv.DictReader(open(sys.argv[1], newline="")))
print(json.dumps({
    "row_count": len(rows),
    "columns": list(rows[0].keys()) if rows else [],
    "paid_rows": sum(1 for row in rows if row["status"] == "paid")
}, ensure_ascii=False))
`,
  );
  await sandbox.files.write(
    "/tmp/agent-harness/tools/validate_business_rule.py",
    `import csv, json, sys
violations = []
for row in csv.DictReader(open(sys.argv[1], newline="")):
    if row["status"] == "paid" and int(row["amount"]) <= 0:
        violations.append({"order_id": row["order_id"], "reason": "paid order amount must be positive"})
print(json.dumps({
    "passed": len(violations) == 0,
    "violations": violations,
    "recommendation": "block_auto_import" if violations else "ready_for_import"
}, ensure_ascii=False))
`,
  );

  const trace = [];
  for (const name of ["profile_dataset", "validate_business_rule"]) {
    const tool = tools[name];
    const result = await sandbox.commands.run(tool.command, {
      cwd: tool.cwd,
      timeoutMs: tool.timeoutMs,
    });
    if (result.exitCode !== 0) {
      throw new Error(result.stderr || result.stdout);
    }
    trace.push({ tool: name, result: JSON.parse(result.stdout) });
  }

  console.log({ sandboxId: sandbox.sandboxId, trace });
} finally {
  await sandbox.kill();
}
```

## Production recommendations

- The business system must maintain the tool registry. Do not let the model generate arbitrary commands directly.
- Set an independent timeout, input size limit, output size limit, and permission level for every tool.
- The Harness should store the full execution trace, but user-facing output should contain only summaries, key logs, and artifact links.
- High-risk actions must go through human approval, including writes to external systems, message sending, file deletion, and access to sensitive data.
- The sandbox filesystem is task-level temporary state. Long-lived state should be written to an external database, object storage, or the business system.

## Runnable example

This practice maps to `npm run agent-harness` in the sample project. The script runs two tool calls inside FC Agent Sandbox, records the trace, and outputs the final business decision.
