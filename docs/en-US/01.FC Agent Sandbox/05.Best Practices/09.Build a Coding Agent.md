# Build a Coding Agent

The business value of a Coding Agent is not that it can generate code. It is that it can finish the change, run validation, and return reviewable results to the developer. Without an isolated execution environment, an Agent can only stop at suggestions. As soon as it needs to install dependencies, run tests, start services, or generate patches, you run into security, resource, and environment consistency issues.

FC Agent Sandbox is a good remote execution layer for a Coding Agent. The main business system handles model invocation, task orchestration, permission control, and result presentation. FC Agent Sandbox handles repository state, command execution, temporary files, logs, and diffs. Destroy the sandbox after the task completes so dependencies, processes, and intermediate files do not pollute the main system.

## Use cases

- A coding assistant reproduces an issue in an isolated environment, edits files, runs tests, and returns a diff.
- A code review system validates whether an AI-generated fix really passes tests before merge.
- An App Builder generates a project and then runs build and preview steps in the sandbox without consuming business service resources.
- An internal enterprise Agent supports multiple language repositories by fixing toolchains such as Python, Node.js, and Go through templates.

## Solution design

A verifiable Coding Agent task is best split into five steps:

1. Create FC Agent Sandbox and write the task ID, repository address, and minimum required environment variables into it.
2. Prepare the workspace by cloning the repository or writing the files to be processed.
3. Run baseline validation to confirm the issue can be reproduced.
4. Write the Agent-generated changes and rerun tests or builds.
5. Return test results, a log summary, and the diff, then release the sandbox.

The key here is "fail first, then fix, then pass". Only when the validation chain is closed should the Agent output move into code review or automated merge flows.

## Example code

The following example simulates a Coding Agent fixing a Python function: it first runs tests and gets a failure, then writes the fixed code, and finally returns the diff.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("code-interpreter-v1", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "coding-agent" },
});

try {
  await sandbox.files.makeDir("/tmp/coding-agent");
  await sandbox.files.write("/tmp/coding-agent/calculator.py", "def add(a, b):\n    return a - b\n");
  await sandbox.files.write(
    "/tmp/coding-agent/test_calculator.py",
    `import unittest
from calculator import add

class CalculatorTest(unittest.TestCase):
    def test_add(self):
        self.assertEqual(add(2, 3), 5)

if __name__ == "__main__":
    unittest.main()
`,
  );

  await sandbox.commands.run("git init && git add .", { cwd: "/tmp/coding-agent" });

  await sandbox.commands.run("python3 -m unittest -v > before.out 2>before.err; echo $? > before.exit", {
    cwd: "/tmp/coding-agent",
    timeoutMs: 30_000,
  });
  const beforeExit = await sandbox.files.read("/tmp/coding-agent/before.exit");

  await sandbox.files.write("/tmp/coding-agent/calculator.py", "def add(a, b):\n    return a + b\n");

  const after = await sandbox.commands.run("python3 -m unittest -v", {
    cwd: "/tmp/coding-agent",
    timeoutMs: 30_000,
  });
  if (after.exitCode !== 0) {
    throw new Error(after.stderr || after.stdout);
  }

  const diff = await sandbox.commands.run("git diff -- calculator.py", {
    cwd: "/tmp/coding-agent",
  });

  console.log({
    beforeExitCode: Number(beforeExit.trim()),
    afterExitCode: after.exitCode,
    patch: diff.stdout,
  });
} finally {
  await sandbox.kill();
}
```

## Production recommendations

- Do not let the model concatenate arbitrary Shell commands. Wrap actions such as dependency installation, test execution, and diff generation into controlled steps.
- Set both a sandbox timeout and command timeouts for every task. Coding Agents often hit long builds, stuck tests, or services that never exit.
- Layer the output: show users the conclusion, failed tests, and diff; send full logs to the audit system.
- Private repositories, package manager tokens, and cloud credentials should all use least privilege and be injected per task instead of being baked into templates.
- In production, maintain custom templates for common tech stacks to reduce dependency installation time on every task.

## Runnable example

This practice maps to `npm run coding-agent` in the sample project. The script creates FC Agent Sandbox, reproduces a failing test, writes the fix, reruns validation, and outputs the diff.
