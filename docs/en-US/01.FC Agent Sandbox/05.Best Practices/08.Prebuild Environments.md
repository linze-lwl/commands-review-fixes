# Prebuild Environments

Agent tasks are sensitive to the runtime environment. Installing system tools, language dependencies, and business SDKs after every task starts causes three problems: slow startup, more failure points, and less control over versions. For scenarios such as Coding Agents, data analysis, document processing, and test validation, custom templates are foundational infrastructure for both stability and cost control.

The best practice is not to stuff every dependency into an image. It is to move stable, reusable, low-change runtime components into the template, while keeping user data, task input, and short-lived credentials injected only after the sandbox is created. After publishing a template, immediately create a sandbox for smoke validation, verify that the toolchain works, and only then switch business traffic.

## Use cases

- A Coding Agent needs fixed versions of Python, Node.js, `git`, build tools, and test tools.
- A data analysis Agent needs commonly used scientific libraries, spreadsheet libraries, and plotting libraries preinstalled.
- An internal enterprise Agent needs business SDKs, root certificates, CLI tools, and a base directory layout preinstalled.
- Multi-tenant tasks need fast startup and cannot spend time repeatedly downloading dependencies.

## Solution design

Template governance is best split into three steps:

1. Build the template with only stable dependencies, system tools, and shared startup scripts.
2. Run smoke validation by creating FC Agent Sandbox from that template and checking critical commands, directories, environment variables, and business entry points.
3. Switch versions only after validation passes; if validation fails, keep business tasks on the old template.

This turns the template into a releasable engineering asset instead of a one-off Dockerfile.

## Example code

The following example creates FC Agent Sandbox from a specific template and checks the base tools required for Agent tasks. In production, replace `PRACTICE_TEMPLATE` with your own template name.

```javascript
import { Sandbox } from "e2b";

const template = process.env.PRACTICE_TEMPLATE || "code-interpreter-v1";
const sandbox = await Sandbox.create(template, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "template-smoke" },
});

try {
  const checks = [
    ["python", "python3 --version"],
    ["git", "git --version"],
    ["workdir", "test -d /tmp && echo tmp-ok"],
  ];

  const results = [];
  for (const [name, command] of checks) {
    const result = await sandbox.commands.run(command, { timeoutMs: 30_000 });
    if (result.exitCode !== 0) {
      throw new Error(`${name} check failed: ${result.stderr || result.stdout}`);
    }
    results.push({ name, output: result.stdout.trim() || result.stderr.trim() });
  }

  console.log({ template, checks: results });
} finally {
  await sandbox.kill();
}
```

## Build recommendations

When building a custom template, keep these boundaries in mind:

- Good candidates for the template: language runtimes, package managers, stable dependencies, basic CLIs, business SDKs, and non-sensitive configuration.
- Not suitable for the template: user data, task input, production secrets, short-lived tokens, and fast-changing business state.
- Pin image versions whenever possible. In production, avoid depending on drifting `latest` tags.
- Every template should have a minimal smoke test and should be released to business tasks only after validation passes.

## Runnable example

This practice maps to `npm run template-smoke` in the sample project. It validates `code-interpreter-v1` by default, and you can also specify a custom template through `PRACTICE_TEMPLATE`.
