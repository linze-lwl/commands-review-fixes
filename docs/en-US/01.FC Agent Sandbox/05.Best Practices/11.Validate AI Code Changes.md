# Validate AI Code Changes

Once AI generates a patch, the most underestimated cost is validation. Code that looks reasonable may still fail tests, and code that passes in a local environment may still fail in a clean one. Running validation inside FC Agent Sandbox ensures every AI code change is verified in an isolated, disposable, and traceable environment.

This practice fits coding assistants, code review bots, automated repair systems, and pre-CI validation gates. FC Agent Sandbox is not a replacement for CI. It provides an earlier validation gate that is closer to the Agent execution flow: as soon as the patch is generated, run tests and feed failure reasons back to the Agent or the developer.

## Use cases

- After an AI-generated fix, automatically run a minimal test set before sending the change to code review.
- A code review bot asks the Agent to continue iterating when tests fail.
- Validate multiple candidate patches in parallel and return only the ones that pass.
- Run validation for untrusted repositories or user code without polluting the main CI worker.

## Solution design

Split the validation flow into three stages:

1. Prepare a clean workspace and write the baseline code, test files, and candidate patch.
2. Run the test command and capture the exit code, `stdout`, `stderr`, and key report files.
3. Based on the test result, decide whether to continue iterating, send the patch for manual review, or discard it.

The validation system should use the command exit code and structured test reports as the decision basis instead of relying only on the model's interpretation of logs.

## Example code

The following example first validates an incorrect discount rule, then writes the AI-generated fix, and finally uses the test result as the merge gate.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("code-interpreter-v1", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "ai-change-validation" },
});

try {
  await sandbox.files.makeDir("/tmp/change-validation");
  await sandbox.files.write(
    "/tmp/change-validation/discount.py",
    `def final_price(amount, vip=False):
    rate = 0.8 if vip else 1.0
    return round(amount * rate, 2)
`,
  );
  await sandbox.files.write(
    "/tmp/change-validation/test_discount.py",
    `import unittest
from discount import final_price

class DiscountTest(unittest.TestCase):
    def test_vip_discount(self):
        self.assertEqual(final_price(100, vip=True), 90)

    def test_regular_price(self):
        self.assertEqual(final_price(100), 100)

if __name__ == "__main__":
    unittest.main()
`,
  );

  await sandbox.commands.run("python3 -m unittest -v > first.out 2>first.err; echo $? > first.exit", {
    cwd: "/tmp/change-validation",
    timeoutMs: 30_000,
  });
  const firstExit = await sandbox.files.read("/tmp/change-validation/first.exit");

  await sandbox.files.write(
    "/tmp/change-validation/discount.py",
    `def final_price(amount, vip=False):
    rate = 0.9 if vip else 1.0
    return round(amount * rate, 2)
`,
  );

  const secondRun = await sandbox.commands.run("rm -rf __pycache__ && python3 -m unittest -v", {
    cwd: "/tmp/change-validation",
    timeoutMs: 30_000,
  });
  if (secondRun.exitCode !== 0) {
    throw new Error(secondRun.stderr || secondRun.stdout);
  }

  console.log({
    firstRunExitCode: Number(firstExit.trim()),
    secondRunExitCode: secondRun.exitCode,
    gate: "pass",
  });
} finally {
  await sandbox.kill();
}
```

## Production recommendations

- Make the test command configurable so different repositories can map to different validation strategies.
- For large repositories, use templates with preinstalled base toolchains to reduce dependency setup time before validation.
- Store the exit code, failed test cases, log summary, and patch version for every validation result so later audits are possible.
- Apply least-privilege controls to network access, private dependencies, and environment variables.
- Do not run a full long-lived CI system inside FC Agent Sandbox. It is better suited to task-level, interactive, Agent-driven validation.

## Runnable example

This practice maps to `npm run verify-change` in the sample project. The script runs a failing test inside FC Agent Sandbox, applies the fix, and reruns validation.
