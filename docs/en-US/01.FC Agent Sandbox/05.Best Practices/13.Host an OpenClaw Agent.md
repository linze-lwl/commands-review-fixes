# Host an OpenClaw Agent

OpenClaw is an open source personal AI assistant. Its core form is not a one-off tool script but a long-running gateway that connects the model, message entry points, tools, Skills, sessions, and a control UI. The official OpenClaw workflow recommends `openclaw onboard` for configuration, `openclaw gateway` for gateway management, and `openclaw dashboard` for opening the console. You can also interact with the Agent directly through CLI commands such as `openclaw agent` and `openclaw message`.

FC Agent Sandbox is a good fit for hosting this kind of complete Agent application in an isolated cloud environment. Preinstall OpenClaw and its dependencies in a custom template, create sandbox instances on demand to run the gateway, expose the control UI through `getHost(port)`, and use `commands.run()` or `Sandbox.connect()` for CLI operations. This lets developers host OpenClaw in the cloud in a way that is close to E2B while keeping the runtime isolated for demos, evaluations, development environment hosting, or internal pilots.

## Use cases

- A team wants to quickly create an isolated OpenClaw Agent instance for customer demos, POCs, or internal trials.
- Developers want to manage OpenClaw through a Web UI while keeping CLI operations available.
- An enterprise wants to evaluate third-party Skills, messaging channels, or tool calls without running them directly on a laptop or a long-lived business host.
- The Agent needs to process files, run commands, access a browser, or call external tools, and requires an isolated boundary with a disposable lifecycle.

## Recommended architecture

Use a "template-pinned software stack, sandbox runtime, and external state" architecture:

- **Custom template**: preinstall Node.js, the OpenClaw CLI, browser dependencies, common tools, and startup scripts.
- **FC Agent Sandbox**: each sandbox runs one OpenClaw gateway instance and hosts the control UI, CLI commands, and Agent tasks.
- **External systems**: store long-lived configuration, audit logs, user identity, business databases, and sensitive credentials.

Do not bake long-lived credentials, production messaging channel tokens, or private enterprise data into the template. Put only software and base dependencies into the template, then inject short-lived credentials at runtime through environment variables or the business-side secret system.

## Build an OpenClaw template

First prepare a base image that includes OpenClaw and push it to an image registry that your FC Agent Sandbox template build can access. The following Dockerfile shows only the key steps. In a real image, pin versions, enable image scanning, and add browser and font dependencies according to enterprise security requirements.

```dockerfile
FROM node:24-bookworm

RUN apt-get update \
  && apt-get install -y --no-install-recommends \
    ca-certificates curl git python3 python3-pip ripgrep \
  && rm -rf /var/lib/apt/lists/*

RUN npm install -g openclaw@latest

WORKDIR /workspace
EXPOSE 18789
```

After building and pushing the image, create an FC Agent Sandbox template through `Template.build()` in the E2B SDK:

```python
import os
import time
from e2b import Template, default_build_logger

image = os.environ["OPENCLAW_IMAGE"]

build = Template.build(
    Template().from_image(image),
    name=f"openclaw-agent-{int(time.time())}",
    cpu_count=2,
    memory_mb=4096,
    api_key=os.environ["E2B_API_KEY"],
    api_url=os.environ["E2B_API_URL"],
    domain=os.environ["E2B_DOMAIN"],
    on_build_logs=default_build_logger(),
)

print("template_id:", build.template_id)
```

## Start the OpenClaw gateway

Use the template ID from the previous step when creating FC Agent Sandbox. For the first run, complete configuration through the CLI, confirm that the model, gateway, control UI, and tool capabilities all work, and only then solidify the flow into startup scripts.

```typescript
import { Sandbox } from "e2b";

const template = process.env.OPENCLAW_TEMPLATE_ID;
if (!template) {
  throw new Error("OPENCLAW_TEMPLATE_ID is required");
}

const sandbox = await Sandbox.create(template, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: {
    OPENCLAW_HOME: "/workspace/.openclaw",
    OPENCLAW_STATE_DIR: "/workspace/.openclaw/state",
    // Inject model or subscription credentials according to OpenClaw requirements.
    // Do not write real credentials into the template or the source repository.
  },
  metadata: {
    app: "openclaw",
    purpose: "agent-demo",
  },
});

try {
  await sandbox.commands.run("mkdir -p /workspace/.openclaw/state");

  const gateway = await sandbox.commands.run(
    "openclaw gateway --port 18789 --verbose",
    {
      cwd: "/workspace",
      background: true,
      timeoutMs: 30_000,
    },
  );

  const health = await sandbox.commands.run("openclaw gateway status", {
    cwd: "/workspace",
    timeoutMs: 30_000,
  });
  console.log(health.stdout);

  const host = sandbox.getHost(18789);
  console.log(`OpenClaw Control UI: https://${host}`);
  console.log(`Gateway process id: ${gateway.pid}`);
} finally {
  // In demo environments you can keep the sandbox for user access.
  // Release it after automated tests finish.
  // await sandbox.kill();
}
```

If `openclaw gateway status` reports that the gateway has not been configured yet, enter the sandbox and run `openclaw onboard`, or use a configuration script to write the gateway, model, and control UI settings. OpenClaw Onboard guides you through model provider selection, API key configuration, gateway setup, and Skills setup. In production, turn those steps into auditable initialization scripts instead of relying on manual interaction during sandbox startup.

## Access the Web UI

The OpenClaw gateway listens on port `18789` by default. After the service starts, obtain the public access domain through `sandbox.getHost(18789)`:

```typescript
const url = `https://${sandbox.getHost(18789)}`;
console.log(url);
```

Before exposing the control UI, confirm the following:

- The gateway listens only on the port that needs to be exposed, and no extra services are opened publicly.
- The control UI is protected by authentication or a shared secret instead of exposing an unauthenticated console.
- If the sandbox is used for demos or evaluation, set a clear lifecycle and cleanup policy.
- For long-term access, use a business gateway, identity system, and audit system to handle production traffic instead of treating the temporary domain returned by `getHost()` as a production domain.

## Manage the Agent through the CLI

The Web UI is better for interactive management, while the CLI is better for automation. You can run the OpenClaw CLI inside FC Agent Sandbox through `commands.run()`:

```typescript
const status = await sandbox.commands.run("openclaw gateway status", {
  cwd: "/workspace",
});

const reply = await sandbox.commands.run(
  `openclaw agent --message "Inspect the current workspace and suggest executable next steps" --thinking high`,
  {
    cwd: "/workspace",
    timeoutMs: 120_000,
  },
);

console.log(status.stdout);
console.log(reply.stdout);
```

If the business service needs to continue managing the same OpenClaw instance across requests, save `sandbox.sandboxId` and reconnect later through `Sandbox.connect()`:

```typescript
const sandboxId = process.env.OPENCLAW_SANDBOX_ID;
if (!sandboxId) {
  throw new Error("OPENCLAW_SANDBOX_ID is required");
}

const sandbox = await Sandbox.connect(sandboxId, {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
});

const result = await sandbox.commands.run("openclaw gateway status", {
  cwd: "/workspace",
});
console.log(result.stdout);
```

## Best practices

- **Validate the CLI first, then expose the Web UI**: first make `openclaw gateway status` and `openclaw agent --message` work, and only then expose the control UI through `getHost()`.
- **Keep only software in the template**: put Node.js, OpenClaw, browser dependencies, and common tools in the template; inject model keys, messaging platform tokens, and enterprise credentials at runtime.
- **Treat the sandbox as disposable runtime**: keep long-term memory, audit logs, and business state in external storage instead of relying on the sandbox filesystem for persistence.
- **Restrict message entry points and public exposure**: OpenClaw can connect to Telegram, Slack, Discord, WhatsApp, and other channels. During pilot phases, expose only low-risk channels and apply access control to DMs, group chats, and unknown senders.
- **Isolate third-party Skills**: Skills installed from the community should be treated as untrusted code. Validate their behavior, permissions, and logs in a temporary sandbox before moving them into production.
