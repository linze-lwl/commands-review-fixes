# Safely Execute Agent Tool Calls

Agent tool calls are the critical path for connecting model capabilities to a business system. The model may decide to call `run_tests`, `analyze_csv`, `convert_file`, or an internal diagnostic tool, but the real risk appears during execution: parameters may be untrusted, input files may be malformed, tool scripts may block, and the result still needs to be audited.

FC Agent Sandbox is a good isolation layer for tool execution. The business system is responsible for model invocation, tool selection, parameter validation, and permission checks. FC Agent Sandbox is responsible for running the tool, enforcing time limits, storing temporary files, and returning structured results. This turns "the model can call tools" into a controlled engineering workflow.

## Use cases

- A coding assistant calls testing, build, or static analysis tools.
- A data Agent calls CSV analysis, data quality, or report generation tools.
- An office Agent calls document conversion, spreadsheet processing, or file parsing tools.
- An operations Agent calls read-only diagnostic scripts and returns structured troubleshooting results.

## Solution design

Tool calls should not be treated as arbitrary command execution. The recommended design is:

1. The model outputs only the tool name and parameters.
2. The business orchestrator resolves that name through a whitelist to a fixed tool entry point.
3. The orchestrator validates parameters, tenant permissions, input files, and timeout settings.
4. FC Agent Sandbox runs the tool and returns JSON results, an exit code, and a log summary.

This design limits model uncertainty to "which tool to select" and "which parameters to provide", instead of letting the model control the runtime directly.

## Example code

The following example executes order analysis as a controlled tool. The model only requests `summarize_orders`; the actual command is determined by a business-side whitelist.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("code-interpreter-v1", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "agent-tool-call" },
});

const tools = {
  summarize_orders: {
    command: "python3 tools/summarize_orders.py data/orders.csv",
    cwd: "/tmp/agent-tool-call",
    timeoutMs: 30_000,
  },
};

function validateToolCall(call) {
  if (!tools[call.name]) {
    throw new Error(`Tool is not allowed: ${call.name}`);
  }
  if (call.arguments && Object.keys(call.arguments).length > 0) {
    throw new Error("Unexpected tool arguments");
  }
}

try {
  await sandbox.files.makeDir("/tmp/agent-tool-call");
  await sandbox.files.makeDir("/tmp/agent-tool-call/data");
  await sandbox.files.makeDir("/tmp/agent-tool-call/tools");
  await sandbox.files.write(
    "/tmp/agent-tool-call/data/orders.csv",
    "order_id,customer,amount,status\n1001,Acme,1200,paid\n1002,Acme,900,paid\n1003,Globex,300,refunded\n",
  );
  await sandbox.files.write(
    "/tmp/agent-tool-call/tools/summarize_orders.py",
    `import csv, json
paid_total = 0
with open("data/orders.csv", newline="") as f:
    for row in csv.DictReader(f):
        if row["status"] == "paid":
            paid_total += int(row["amount"])
print(json.dumps({"paid_total": paid_total, "business_summary": f"Paid revenue is {paid_total}."}))
`,
  );

  const call = { name: "summarize_orders", arguments: {} };
  validateToolCall(call);

  const result = await sandbox.commands.run(tools[call.name].command, {
    cwd: tools[call.name].cwd,
    timeoutMs: tools[call.name].timeoutMs,
  });
  if (result.exitCode !== 0) {
    throw new Error(result.stderr || result.stdout);
  }

  console.log(JSON.parse(result.stdout));
} finally {
  await sandbox.kill();
}
```

## Production recommendations

- Keep tool entry points fixed, validate parameters explicitly, and never concatenate model output into a Shell command.
- Set an independent timeout, input size limit, and output size limit for each tool.
- Prefer JSON for tool results and send full logs to the audit system.
- Inject credentials per task with short lifetimes and least privilege, and do not bake them into the template.
- Add human approval or a policy engine for high-risk tools such as file deletion, external network access, or write operations against cloud resources.

## Runnable example

This practice maps to `npm run tool-call` in the sample project. The script runs a controlled order-analysis tool inside FC Agent Sandbox and returns a structured business result.
