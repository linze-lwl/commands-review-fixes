# Run MCP Tool Services

The value of MCP is that it standardizes tool discovery, parameter descriptions, invocation entry points, and result formats. For enterprise Agents, that reduces tool integration cost and makes it easier to expose internal capabilities to Agents through a protocol. The boundary matters, though: this document describes how to run custom MCP-style tool services inside FC Agent Sandbox, not how FC Agent Sandbox supports the E2B MCP Gateway.

A safer implementation pattern is to place the Agent Runner and tool service inside the same FC Agent Sandbox and call the service through `localhost`. This lets you use an MCP-style tool abstraction without exposing the tool service directly to the public internet. When the task ends, destroy the sandbox so the tool service and temporary state are released together.

## Use cases

- Start a project-specific tool service for a single Agent task and destroy it when the task ends.
- Wrap internal enterprise tools behind MCP-style interfaces while keeping their runtime isolated from the main business service.
- Run the Agent Runner, tool service, and temporary file-processing logic inside the same sandbox.
- Isolate execution for tool parameters and input files that are not fully trusted.

## Solution design

The recommended flow is:

1. The business system creates FC Agent Sandbox and writes both the tool service and the Agent Runner into it.
2. Start the tool service as a background command and have it listen on `127.0.0.1`.
3. Let the Agent Runner call the tool service through `localhost` inside the same sandbox.
4. The tool service returns structured results, and the Runner aggregates the output.
5. The business system reads the result, stops the background process, and releases the sandbox.

This pattern is suitable for task-level tool services, not for a long-running online API gateway. If external MCP Clients must access the service directly from outside the sandbox, public access, authentication, session management, and access control should be designed as separate product concerns.

## Example code

The following example starts a JSON-RPC tool service inside the sandbox, exposes a `score_lead` tool, and calls it from a client running in the same sandbox.

```javascript
import { Sandbox } from "e2b";

const sandbox = await Sandbox.create("code-interpreter-v1", {
  apiKey: process.env.E2B_API_KEY,
  apiUrl: process.env.E2B_API_URL,
  domain: process.env.E2B_DOMAIN,
  envs: { TASK_TYPE: "mcp-tool-service" },
});

const server = `from http.server import BaseHTTPRequestHandler, HTTPServer
import json

def score_lead(arguments):
    employees = int(arguments.get("employees", 0))
    intent = arguments.get("intent", "low")
    score = min(100, employees // 10 + (40 if intent == "high" else 10))
    return {"content": [{"type": "text", "text": json.dumps({
        "score": score,
        "tier": "enterprise" if score >= 70 else "growth",
        "next_action": "route_to_sales" if score >= 70 else "nurture"
    })}]}

class Handler(BaseHTTPRequestHandler):
    def do_POST(self):
        length = int(self.headers.get("content-length", "0"))
        request = json.loads(self.rfile.read(length))
        method = request.get("method")
        params = request.get("params") or {}
        if method == "tools/call":
            result = score_lead(params.get("arguments") or {})
        else:
            result = {"ok": True}
        body = json.dumps({"jsonrpc": "2.0", "id": request.get("id"), "result": result}).encode()
        self.send_response(200)
        self.send_header("content-type", "application/json")
        self.send_header("content-length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

    def log_message(self, format, *args):
        return

HTTPServer(("127.0.0.1", 8765), Handler).serve_forever()
`;

const client = `import json, urllib.request
body = json.dumps({
    "jsonrpc": "2.0",
    "id": "tools/call",
    "method": "tools/call",
    "params": {"name": "score_lead", "arguments": {"employees": 820, "intent": "high"}}
}).encode()
request = urllib.request.Request("http://127.0.0.1:8765", data=body, headers={"content-type": "application/json"})
print(urllib.request.urlopen(request, timeout=5).read().decode())
`;

let serverCommand;

try {
  await sandbox.files.makeDir("/tmp/mcp-tool-server");
  await sandbox.files.write("/tmp/mcp-tool-server/server.py", server);
  await sandbox.files.write("/tmp/mcp-tool-server/client.py", client);

  serverCommand = await sandbox.commands.run("python3 server.py", {
    cwd: "/tmp/mcp-tool-server",
    background: true,
  });

  const result = await sandbox.commands.run("python3 client.py", {
    cwd: "/tmp/mcp-tool-server",
    timeoutMs: 30_000,
  });
  if (result.exitCode !== 0) {
    throw new Error(result.stderr || result.stdout);
  }

  console.log(JSON.parse(result.stdout));
} finally {
  if (serverCommand?.pid) {
    await sandbox.commands.kill(serverCommand.pid);
  }
  await sandbox.kill();
}
```

## Production recommendations

- Prefer local calls between the Agent Runner and the tool service inside the same sandbox to reduce public exposure.
- Register only the tools required for the current task instead of exposing internal capabilities to the model all at once.
- `inputSchema` describes parameters only. The business side still needs strong validation.
- Background tool services must be terminated when the task ends to avoid leftover processes.
- Do not describe this pattern as a hosted MCP Gateway. A service catalog, centralized authentication, and cross-service routing require separate product capabilities.

## Runnable example

This practice maps to `npm run mcp` in the sample project. The script starts a tool service inside FC Agent Sandbox and completes the call through a sandbox-local client.
